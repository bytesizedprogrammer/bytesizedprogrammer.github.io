Project Details:

Software you'll need for this project to run before following instructions:
  • A slippi account and correct version of dolphin emulator as can be obtained on slippi.gg
  • Your own legally obtained copy of Super Smash Bros. Melee (We do not condone piracy in the slightest)
  • MySQL Server 8.4 (I recommend also downloading MySQL Workbench 8.0 CE as it's my preference, but it's unnecessary)
  • Python 3.12.4
  • Node JS 20.13.1
  • VS Code or alternative (This guide will use VS Code)
  • SlippC

Instructions below on how to run, as it will not be explained here. This project is a project that you run locally on your computer.

The way this project functions is quite simple, but a long process. It starts with SlippC, a program created by a group of 5 developers that converts a singular slp file to JSON. What we did is we took this script and automated it to the take all the slp files in a folder and run the command on all of them. After completion of converting each slp file to JSON, it then automatically deletes all of the slp files you converted. Afterwards, we made another python script that takes said JSON files, and extracts the data we want and sends them up to the MySQL database assuming they fit the criteria expected. What I mean is it looks at each JSON file one at a time, and checks for certain criteria like if the player is on a ban list of players you don't want the matches to count for, if the match was long enough, if the match was completed, etc. and if valid, it then sends the right info up to the database. That's how we achieve the dataset that will come in handy later. It then goes to Slippi-JS, an application developed by Project Slippi that fetches tons of real-time data from your game of SSBM . We took this project (Slippi-JS) and modified it to take note of additional details to prepare it for sending data through websocket to the python server. We did this through various actions such as making the data given through the program to be more easily understandable to the server (such as submitting character names and the stage name instead of ids as this way it is primed for the SQL command), making it understand which player is which, restrictng the amount of times data is sent (as to avoid server overload/data being sent too often and breaking/making the program borderline useless), etc. Afterwards, we addressed the python server, also the file where the ML is run. We first setup the Websocket to take in the data that Slippi-JS was sending so we can receive real time data. After receiving it, we took the data of opponent character and stage and ran the SQL query SELECT * FROM training_data WHERE character_opp = ? AND stage = ? that way we can receive previous match history from the database that we created earlier. What we did next was setup the ML model, and had it get trained off of the database data. We set it up so that 80% of the data would be utilized for training, and the other 20% would be used for testing. We then have the whole ML process restart until the accuracy rate is at least 80%, and then the project is mostly complete. The final step is that the chances of you winning updates about every 8 seconds until the game ends, so we had the ML run with the updated numbers obtain throughout the match so you can see the chances change throughout the whole match.

**Additionally, please note that in a future update, most of this process will be completely automated for you via the desktop app and a .exe file that'll pre-install and pre-setup (almost) everything for the user so that it becomes less complicated and more efficient to run.
